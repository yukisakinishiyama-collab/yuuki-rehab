# 設計上の決定事項

## プロトコル立案機能

### データ永続化
- **決定**: バックエンド DB なしで localStorage のみ使用
- **理由**: 既存アーキテクチャと統一。サーバーレスコストをゼロに抑える。院内 LAN のみでも動作可能にする。

### AI プロトコル生成
- **決定**: `claude-opus-4-5` モデルを使用、最大 4096 トークン
- **理由**: 4 フェーズ × 詳細 JSON は応答量が多い。最新モデルで構造化出力の精度を確保する。

### 同意ゲート
- **決定**: AI モード選択後に患者情報送信前に明示的チェックボックス同意を要求
- **理由**: 患者情報を外部 API（Anthropic）に送信することへのインフォームドコンセントを担保するため。個人情報保護法・医療倫理上の要件。

### フェーズ移行ロジック
- **決定**: 「全移行基準を充足」したときのみ「次フェーズへ進む」ボタンを表示（時間基準なし）
- **理由**: エビデンスベースのリハビリでは時間よりも機能的マイルストーンが推奨されるため。臨床判断を支援するが、自動進行はしない。

### テンプレート照合
- **決定**: キーワードマッチング（正規表現なし、単純 includes）で 5 テンプレートから選択
- **理由**: 入力診断名の揺れ（半角/全角・略称）に対応しつつ、実装を単純に保つ。

### PhaseCard の折りたたみ要素
- **決定**: 外側の折りたたみ領域を `<button>` ではなく `<div role="button">` で実装
- **理由**: HTML 仕様では `<button>` の中に `<button>` を入れてはならない（インタラクティブ要素のネスト禁止）。内部に編集ボタンがあるため `<div role="button">` でラップする。

## 免責事項の表示方針

- **決定**: すべてのプロトコル関連ページの最上部に DisclaimerBanner を表示。AI 生成プロトコルには追加の「AI生成・要臨床確認」バッジを表示。
- **理由**: 医療広告ガイドライン・医師法・薬機法への対応。ツールが医療行為を代替しないことをユーザーに常に認識させるため。

## 患者向けビュー

- **決定**: 「やさしい言葉モード」はクライアントサイドの文字列置換で実装（別データなし）
- **理由**: テンプレートデータを重複管理せずに済む。変換ルールは `PatientView.tsx` 内の Map で一元管理。

## Confetti アニメーション

- **決定**: 外部ライブラリではなく CSS キーフレーム + 動的 DOM 生成で実装
- **理由**: バンドルサイズを増やさずに軽量な祝福演出を実現する。3.5 秒後に自動削除。
