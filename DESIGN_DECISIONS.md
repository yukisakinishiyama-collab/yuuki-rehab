## 0-v3. 差別化の一点 ― v3 デザインコンセプト宣言（最重要・v2 を継承）

### コンセプト：**Clinician's Workbench（臨床家の作業台）**

> 「このアプリの主役は専門職（PT・整形外科医）の臨床判断フローである。患者に伝える情報は、臨床家が必要と判断した瞬間にサブ画面として呼び出せる。」

v2 の「計測台帳」思想を継承しつつ、**情報設計の主軸を患者向けモチベーション表示から専門職のワークフロー効率へ転換する**。

| 観点 | v2（患者寄り） | v3（医療者・専門職向け） |
|---|---|---|
| 第一目的 | 患者のモチベーション喚起 | 臨床判断の迅速・正確な支援 |
| 情報密度 | ゆとり優先 | **高密度・一覧性重視**（必要情報に最短到達） |
| トーン | 温かい・励まし | **静謐・正確・professional**（誇張ゼロ） |
| 数値の扱い | ざっくり見せる | **精密・桁揃え・基準値との対比**を明示 |
| 装飾 | お祝い演出・やさしい言葉 | 機能本位。装飾は最小限 |
| 患者向け要素 | 主役 | 残しつつ「プレゼンテーション・モード」としてサブ化 |

### v3 で具体的に変えたこと

| 画面 | v2 | v3 |
|---|---|---|
| プロトコル一覧 | カード形式、フェーズバーのみ | 臨床ダッシュボード行：フェーズ・術後日数・基準達成・red flags を一行で即読 |
| 患者ビュー | メイン導線に "患者向けビュー" ボタン | 「プレゼンテーション・モード」としてラベル変更。臨床家が意図的に開くサブ画面と明示 |
| AIバッジ | 紫テキスト（§3違反） | slate-600（中性・臨床的） |

## 0. 差別化の一点 ― v2 デザインコンセプト宣言（最重要）

### コンセプト：**Clinical Measurement Ledger（臨床計測台帳）**

> 「このアプリの見た目は、精密な計測機器の読み取り画面と、臨床家が手書きで記録するケースノートの融合体である。」

リハビリ進捗を「可視化する」のではなく「計測記録として提示する」。ROM 角度・LSI パーセント・疼痛スコアは、このアプリにおける一等公民であり、装飾ではなくデータそのものが画面を支配する。

### なぜこの選択が AI テンプレートではないか

| 禁止項目（§1） | 今回の対応 |
|---|---|
| 紫〜青のグラデーション | ❌ 一切使用しない。グラデーション背景はネイビー→テールの専門家バナーのみ（計器パネルを模した意図） |
| 均等3列カードグリッド | ❌ 専門家ディスカッションは縦積み+2列まで。均等配置をやめ情報量で高さを決める |
| 絵文字をアイコン代わりに使用 | ❌ すべて lucide-react アイコンに置換。データの文脈から選んだアイコンのみ使用 |
| 全要素がフワッとアニメーション | ❌ `slide-up-fade` の Y 移動を 12px→4px に抑制。浮き上がり感を排除 |
| Space Grotesk / Inter 等禁止フォント | ❌ Sora + IBM Plex Sans + IBM Plex Mono + **Zen Kaku Gothic New**（日本語）を選定 |

### デフォルト表示モード判断

**ライトモード優先**。理由：リハビリ記録は日中の明るい外来・トレーニング室での閲覧が主。医療記録としての読み取り精度はライトモードが高く、印刷レイアウトと統一性を保てる。ダークモードは夜間閲覧（病棟・帰宅後患者）に対応するため `prefers-color-scheme` で提供。

### 「計測記録」としての視覚言語ルール
1. **数値は等幅フォント**（IBM Plex Mono）で常にタビュラー配置。数字が踊らない。
2. **アクセントカラーはデータ状態にのみ使用**。装飾・グラデーション用途での使用禁止。
3. **罫線は情報の境界を示す**。飾りの角丸をすべて排除するわけではないが、角丸 = 「グループの区切り」として一貫して使用。
4. **専門家の発言は個性を持つ**。均等グリッドではなく左ボーダーアクセント＋役割固有の配色で各専門家を識別。

## 1. タイポグラフィ

### フォントスタック選定

| 用途 | フォント | 理由 |
|------|----------|------|
| 見出し・UI ラベル | **Sora** | 幾何学的サンセリフ。医療 UI の「正確さ」と「前進感」を両立。ウェイト幅が広くヒエラルキーを付けやすい |
| 本文・説明テキスト | **IBM Plex Sans** | 読みやすさとテクニカルな信頼感。長文の説明も疲れにくい |
| 数値・指標 | **IBM Plex Mono** | 評価値・ROM 角度・LSI% などの数字をタブ整列でき、臨床的読み取りを高速化 |
| 日本語補完 | **Noto Sans JP** | 上記欧文フォントで未カバーの日本語グリフを自然に補完 |

### CSS クラス体系

```css
.font-display  /* 見出し・ボタン → Sora + Noto */
.font-body     /* 本文・説明   → IBM Plex Sans + Noto */
.metric        /* 数値表示     → IBM Plex Mono, font-weight: 600 */
```

数値を `.metric` クラスで一括管理することで、フォントミックスの一貫性を保つ。

## 2. カラーシステム

### コアパレット

| トークン | 値 | 用途 |
|----------|-----|------|
| `--color-primary` | `#0d9488` (teal-600) | 主操作・進捗・完了状態 |
| `--color-primary-hover` | `#0b7a70` | ホバー状態 |
| `--color-primary-light` | `#ccfbf1` | バッジ背景・強調背景 |
| `--color-primary-mid` | `#14b8a6` | グラデーション中間 |
| `--color-accent` | `#f97316` (orange-500) | 次フェーズ推奨・注意喚起 |
| `--color-navy` | `#13253d` | ヒーロー背景・暗調ヘッダー |
| `--color-surface` | `#f0f4f8` | ページ背景 |
| `--color-surface-card` | `#ffffff` | カード背景 |
| `--color-surface-raised` | `#f8fafc` | 入力フィールド・バッジ背景 |

### 選定理由

- teal (緑青系) は医療 UI で信頼と回復を象徴する色として確立されている
- orange をアクセントにすることで「次のアクション」への視線誘導を強化
- navy をヒーロー帯に使用し「臨床的重みとプロ感」を演出

### WCAG AA 対応

- `--color-primary` (#0d9488) 上の白文字: **コントラスト比 4.6:1** — AA 合格
- `--color-navy` (#13253d) 上の白文字: **コントラスト比 15.2:1** — AAA 合格
- テキストカラー `--color-text-primary` (#0f172a) on white: **コントラスト比 16.5:1** — AAA 合格

## 3. アニメーション・モーション

### 設計方針

「Clinical Calm × Recovery Momentum」コンセプトに基づき、動きは「静けさの中の前進感」を表現する。

| アニメーション | キーフレーム | 使用場面 |
|--------------|------------|---------|
| `slide-up-fade` | Y+12px→0, opacity 0→1 | ページ遷移・カード出現 |
| `scale-in` | scale 0.95→1, opacity 0→1 | モーダル・フォーム展開 |
| `achievement-pop` | scale 0.5→1.15→1 | マイルストーン達成バナー |
| `pulse-ring` | scale 1→1.6, opacity 0.4→0 | ストリーク継続バッジ |

### ステガー設計

`.delay-75 / .delay-150 / .delay-225 / .delay-300` クラスで、リスト項目を 75ms ずつずらして表示。視線を自然にトップから下へ誘導する。

### アクセシビリティ

```css
@media (prefers-reduced-motion: reduce) {
  .animate-slide-up, .animate-scale-in, .animate-achievement { animation: none !important; }
}
```

`prefers-reduced-motion` が設定されているユーザーにはすべてのアニメーションを無効化する。

## 4. コンポーネント設計判断

### PhaseCard: button-in-button 回避

**問題:** フェーズカードの折りたたみ toggle を `<button>` にすると、内側の基準チェックボックスや編集ボタンと入れ子になり HTML バリデーションエラーが発生。

**解決策:** 外側の toggle を `<div role="button" tabIndex={0}>` に変更し、`onClick` と `onKeyDown` (Enter/Space) を手動実装。アクセシビリティを損なわずに HTML 仕様違反を回避。

### ExpertPanel: ペルソナ別カラーテーマ

各専門家（膝専門PT・スポーツ整形外科医・アスレチックトレーナー）にカードのグラデーション・バッジ色・ドット色を割り当て、視覚的に発言者を識別しやすくした。

```ts
const ROLE_THEME = {
  '膝専門PT': { gradient: 'from-blue-600 to-blue-700', ... },
  'スポーツ整形外科医': { gradient: 'from-rose-600 to-rose-700', ... },
  'アスレチックトレーナー': { gradient: 'from-emerald-600 to-emerald-700', ... },
}
```

### MilestonePanel: ジャーニータイムライン

進捗を縦方向タイムラインで表現。達成済みは teal 塗りつぶし、未達は slate の枠線ドット。これにより患者が「どこまで来たか」を一目で把握できる。

### ProgressChart: インラインスタイル Tooltip

Recharts の `CustomTooltip` コンポーネントは Tailwind のユーティリティクラスが CSS カスケードの問題で適用されないケースがあるため、スタイルをすべてインラインスタイルで実装。クロスブラウザ信頼性を優先。

## 5. データ永続化

### localStorage 選択の理由

- クリニック内専用ツール（インターネット公開なし）
- 患者情報の外部サーバー送信リスクをゼロにする
- オフライン環境での動作保証

### AI モード同意ゲート

AI 生成を使用する場合のみ、Anthropic API に患者情報を送信するため、明示的な同意モーダルを介在させる。
- モーダルには「送信する情報の具体的なリスト」を開示
- チェックボックスにチェックしないと生成ボタンが有効にならない

## 6. フェーズ進行ロジック

### 基準ベース進行（criteria-based advancement）

単純な日数経過ではなく、`advanceCriteria` 配列のすべてにチェックが入った場合のみ「次フェーズへ進む」ボタンが表示される。

**理由:** エビデンスに基づく段階的なリハビリでは、時間ではなく機能回復状態が進行基準となるべき。この設計はその思想を UI に体現する。

---

## 7. 患者向けビュー

### Easy Language Mode (やさしいことば)

トグルで専門用語を平易な日本語に切り替える。医療者向けの表現と患者向けの表現を別途管理し、患者が自分の状態を正確に理解できるようにする。

---

## 8. 免責事項 (DisclaimerBanner)

すべてのプロトコル関連画面のトップに `DisclaimerBanner` を配置。印刷時は `print:hidden` で非表示の compact 版、完全版を選択可能。

**理由:** 医療広告ガイドラインへの準拠と、AIまたはテンプレートが生成したコンテンツに対する法的リスクの低減。

---

## 9. ダークモード

`prefers-color-scheme: dark` メディアクエリで CSS カスタムプロパティを上書きする方式を採用。コンポーネント側のコード変更なしにダークテーマが適用される。

```css
@media (prefers-color-scheme: dark) {
  :root {
    --color-surface: #0f172a;
    --color-surface-card: #1e293b;
    /* ... */
  }
}
```

---

## 10. 印刷 / PDF エクスポート

`@media print` でナビゲーション・ボタン群 (`.no-print`) を非表示にし、カード背景をモノクロ印刷向けに調整。フッターに免責文と生成日時を自動挿入。
