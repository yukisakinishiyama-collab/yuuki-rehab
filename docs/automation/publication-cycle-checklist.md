# 公開サイクル チェックリスト v1.0（全外部公開物の共通ゲート）

作成：Automation AI ／ 作成日：2026-07-12 ／ 状態：CEO 承認待ち
初回運用サイクル（テーマ①・コミット 5bcf51a）で実際に使った検証を再利用可能な形に標準化したもの。
固定フロー（CEO 承認済み）：**制作 → Medical QA 検証 → Legal 審査 → CEO 承認 → 公開**

## 使い方

外部公開物1件ごとに本チェックリストを複製し、各段の担当 AI がチェックする。**全段のチェックが揃うまで次段に進めない。** CEO 承認欄が未記入の公開は禁止（後述「無承認公開を防ぐルール」）。

---

## ① Research AI チェック（エビデンスの土台）

- [ ] 使用する数値・医学的主張がすべて確定リスト（`0X-evidence-summary.md`）に存在する
- [ ] 数値に対象集団・期間・アウトカム定義の条件が付いている
- [ ] 批判的吟味（`0X-critical-appraisal.md`）が存在し、限界が明示されている
- [ ] リスト外の新規数値が含まれていない（あれば Research へ差し戻し）

## ② Marketing AI チェック（表現・チャネル規格）

- [ ] 禁止表現マスターリスト（`10_Legal_AI.md`）の語が含まれていない
- [ ] 効果への言及に「効果には個人差があります」が近接配置されている
- [ ] 「再発予防を目指す／取り組む／確認する」の非断定表現になっている
- [ ] 医科受診案内 or 医療連携の記載がある
- [ ] 院名・エリア（下関市・彦島）が CTA に含まれる
- [ ] チャネル規格（文字数・枚数・体裁）を満たす
- [ ] 恐怖訴求単独の構成になっていない

## ③ Medical QA チェック（医学的正確性・安全性）

- [ ] 主要な数値・主張を原典（または確定リスト＋ガイドライン）と突合した
- [ ] 過大一般化・因果の言い切り・選択的引用がない
- [ ] 運動指導に**中止基準**（「痛みがある場合は行わない」等）がある
- [ ] **受診目安**（強い痛み・強い腫れ・歩行困難・骨折疑い→医療機関）がある
- [ ] 柔道整復師の業務範囲を超える記述（診断確定・リスク判定）がない
- [ ] 見解が分かれる論点が「分かれている」と明示されている
- [ ] 検証結果票（`1X-medical-qa-report.md`）を発行した
- [ ] 未確認事項は「未確認」と明示し公開物から除外した

## ④ Legal AI チェック（適法性・最終ゲート）

- [ ] 禁止表現の機械チェック（grep）を実行し 0 件
- [ ] 最上級・比較優良・保証表現がない
- [ ] 医師・病院・他院への否定・比較がない
- [ ] 保険適用・自賠責の誤認表現がない
- [ ] 体験談・ビフォーアフター・患者出演がない
- [ ] 院独自実績の数値がない（出典つき研究報告のみ）
- [ ] ステマ規制に触れる行為がない（自院名義の発信）
- [ ] ハッシュタグ・テロップ・注記を含む**全文**を審査した
- [ ] 審査結果票（`1X-legal-review-report.md`）を発行した

## ⑤ CEO 承認チェック（人による最終判断・省略不可）

- [ ] 最終承認パケット（全文まとめ）を CEO に提示した
- [ ] CEO の承認記録（承認者・日時・対象版）がある
- [ ] 「承認/修正依頼/却下」が公開物ごとに明示されている
- [ ] 修正依頼の場合、修正後に再審査（③④）を通した

## ⑥ 公開前チェック（公開直前）

- [ ] 公開する版が CEO 承認済みの版と**一字一句一致**する（承認後の無断改変なし）
- [ ] リール等のテロップは最終書き出し版で Legal 再確認済み（Legal L-4 条件）
- [ ] 素材に患者・個人情報の写り込みがない／肖像同意（未成年は保護者）を確認
- [ ] 公開日時・チャネル・公開順が承認どおり
- [ ] kill switch（停止フラグ）が有効で、公開前確認ノードが通っている

## ⑦ 公開後ログ

- [ ] 監査ログに記録（実行ID・入力版のコミットハッシュ・各ゲート結果・承認者・公開日時・URL・スクリーンショット）
- [ ] 公開物のスクリーンショットを保存（`docs/automation/logs/`）
- [ ] 投稿カレンダーの状態を「公開済み」に更新
- [ ] 30日後の指標取得を予約し、BI AI 月次データへ連携

## 修正が必要になった場合の再審査ルール

1. **文言変更は必ず制作段階（①②）に戻す**。「軽微だから」でチェックを飛ばさない
2. 医学的内容に触れる変更 → ③④⑤を再実行。表現のみの変更 → ④⑤を再実行（Legal 判断で③要否を決める）
3. CEO 承認後の変更は**必ず再承認**を取る。承認済み版と異なる版は公開しない
4. 再審査時は版番号を上げ、変更履歴（変更前→変更後→理由）を残す

## 無承認公開を防ぐルール（多重ロック）

1. 公開処理の直前に「CEO 承認記録の存在」と「承認版ハッシュ＝公開版ハッシュ」を機械検証。不一致なら公開を中止
2. 承認期限内に応答がない場合、**無応答は承認とみなさない**（公開せず繰り越し）
3. Automation（L2）は承認記録なしに投稿 API を呼べない構造にする（承認 Webhook が前提条件）
4. kill switch は CEO・Executive の双方が発火可能。発火中は全公開処理を凍結
5. 手動投稿時も本チェックリストの⑤⑥を紙／画面で確認してから実行

## n8n 化する場合のチェック項目

- [ ] 各ゲート（①〜④）を独立ノード化し、結果を構造化出力（JSON: 承認/差し戻し＋指摘）で受け取る
- [ ] ④の後に「機械チェック結果」と「Legal 票」の両方の存在を検証する IF ノード
- [ ] ⑤の承認は Wait（Webhook）ノードで受け、タイムアウト＝繰り越し（承認扱いにしない）
- [ ] 公開ノードの前に「承認版ハッシュ＝公開版ハッシュ」検証ノードと「停止フラグ」確認ノードを直列配置
- [ ] 認証情報は n8n credentials（フロー定義に直書きしない）
- [ ] 全実行を監査ログノードで記録（削除・改変不可の保存先）
- [ ] 失敗時：差し戻しは制作へ Issue 化、API 失敗は Retry 3回→手動切替通知
- [ ] 運用初期12週間は AI ゲート（③④）に人の確認を並行し、乖離を記録
