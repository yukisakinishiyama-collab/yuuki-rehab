# フロー設計書 #002：Google 口コミ返信（自動化レベル L2）

作成：Automation AI ／ 作成日：2026-07-12 ／ 状態：基本方針 CEO 承認済み（2026-07-12）・テンプレートは承認待ち
自動化レベル：**L2（AI 下書き → CEO 承認 → 実行）**。承認記録なしに返信できない構造とする。

## CEO 決定（2026-07-12・初期運用で固定）

- **完全自動返信は禁止**（初期運用）
- 肯定的口コミ：AI 下書き → CEO 一括承認 → 返信
- 中立口コミ：AI 下書き → CEO 確認 → 返信
- ネガティブ口コミ：自動返信禁止・L3 手動対応
- 医療内容・金銭・保険・事故・クレームを含む口コミ：自動返信禁止・L3 手動対応
- 口コミ本文の症状・個別事情を返信で繰り返さない／来院事実を院側から過度に確認しない
- 返信は「感謝・一般的な方針・今後の丁寧な対応」のみ
- 返信テンプレートは `docs/automation/review-reply-template-pack.md` を使用
関連：`Yuki_AI_OS/05_MEO_AI.md`（口コミ運用原則）／`10_Legal_AI.md`／`gbp-posting-l2-flow.md`（#001）

## 目的

Google 口コミへの返信を48時間以内目標で安定運用し、感謝と信頼を可視化する。同時に、個人情報・症状詳細・広告規制違反となる返信を構造的に防止する。

## 対象業務

- GBP に投稿された新規口コミの検知 → 分類 → 返信ドラフト生成 → 承認 → 返信 → 記録
- 対象外：口コミの削除申請（ポリシー違反口コミは手動で Google へ報告・L3）、口コミ依頼施策（14_Offline の口コミカード）

## トリガー

- 定時ポーリング：1日1回（朝）GBP の新規口コミを確認
- 将来：GBP API の通知（Webhook）に置換

## 口コミ取得方法

- 当面：GBP 管理画面の手動確認（CEO または受付）→ 内容を入力票に転記
- n8n 実装後：Google Business Profile API で新規レビューを取得（星数・本文・投稿者表示名・日時）
- 取得した口コミ本文は監査ログに保存（返信判断の記録として）

## 分類（返信ドラフト生成の前段）

| 分類 | 条件 | 扱い |
|---|---|---|
| A：肯定（星4-5・通常内容） | 症状・施術への具体的言及なし〜軽微 | 定型ベースで L2 返信 |
| B：肯定（症状に具体的言及あり） | 「捻挫が良くなった」等の効果言及を含む | L2 返信（効果への同調をしない定型） |
| C：中立・要望（星3前後） | 待ち時間・設備等への要望 | L2 返信（感謝＋改善姿勢）。内容により CEO 相談 |
| D：ネガティブ（星1-2・不満/クレーム） | 苦情・トラブル・事実関係の争い | **手動対応（L3）へ即切替**。ドラフトは参考情報として作成可 |
| E：ポリシー違反疑い | 誹謗中傷・無関係・なりすまし・個人攻撃 | 返信せず Google へ報告を検討（L3・CEO 判断） |

## 返信文生成ルール

1. 構成は3部固定：**感謝 → 一般的情報（任意）→ 締め**。80〜200字目安
2. 使用できる文型はテンプレート集（承認済み）からのみ選択。テンプレート外の新規文章が必要な場合は Marketing AI へ差し戻し
3. 投稿者の表示名は「〇〇様」形式でのみ使用（本名らしき場合は「ご投稿ありがとうございます」で名指しを避ける）
4. 院側から症状・来院日・施術内容に言及しない（下記ルール）
5. 医療的アドバイス・効果の約束・来院の勧誘（「またのご来院をお待ちしています」程度は可、施術予約の営業は不可）を書かない

### テンプレート例（初期セット・Legal 確認対象）

- A用：「ご来院とあたたかい口コミをありがとうございます。今後も安心してご利用いただけるよう努めてまいります。」
- B用：「ご投稿ありがとうございます。経過に合わせて無理のないよう取り組んでいただけるよう、今後もサポートしてまいります。※個別の状態への言及は控えております。」
- C用：「貴重なご意見をありがとうございます。いただいた内容は院内で共有し、改善に努めてまいります。」

## 個人情報・症状詳細を書かないルール（最重要）

- **来院の事実を確認する記述をしない**（「先日はご来院ありがとうございました」は、投稿者が患者であることを院が認める記述になるため、投稿文側に来院言及がある場合のみ可。判断に迷えば「ご投稿ありがとうございます」へ）
- 症状名・部位・経過・施術内容・担当者名を院側から書かない（投稿者が書いていても**繰り返さない**）
- 年齢・学校名・競技名等、本人特定につながる情報に触れない
- 守秘義務（柔道整復師法）と個人情報保護法を返信にも適用する

## ネガティブ口コミ（分類D）の対応

1. **自動返信しない**。検知後ただちに CEO へ通知（L3）
2. 対応原則：反論しない／事実関係を口コミ上で争わない／感謝と改善姿勢を示す／必要に応じ「詳細は直接ご連絡ください」と院の連絡先（公開情報のみ)を案内
3. 事実誤認・悪質な内容は返信より先に Google への報告を検討（Legal AI が要件確認）
4. 返信文は CEO と Legal AI の両承認後にのみ投稿
5. 返信後も再燃した場合は追加返信せず、L3（炎上対応）へ移行

## Medical QA が必要になる条件

- 返信に医学的内容（症状・運動・経過への言及）を含める場合 → 原則含めないため通常は不要。含める例外時は Medical QA 検証票を必須とする
- 口コミ本文が医学的に誤った情報を広めている場合（例：「捻挫は冷やせば治ると教わった」）→ 返信で訂正はせず、Medical QA と協議のうえ一般情報の発信（GBP 投稿・Q&A）で対応

## Legal AI が必要になる条件

- 分類 D・E のすべて（返信文・報告要否の判断）
- テンプレート集の新規追加・改訂時
- 返信に院の制度的説明（保険・料金・交通事故対応）を含める場合
- 投稿者から法的措置の示唆・返金要求等がある場合（→ L3、弁護士相談の要否を CEO に進言）

## CEO 承認フロー

- 分類 A〜C：日次でまとめて承認依頼（ドラフト＋口コミ原文＋分類）。承認期限：ドラフト提示から48時間（期限超過は翌日再提示。無応答=承認としない）
- 分類 D・E：即時個別通知。CEO＋Legal の両承認まで返信しない
- 当面の承認手段：GitHub Issue または本リポジトリの承認ファイルに **CEO_INPUT_REQUIRED** として記録（LINE 接続後に移行）

## 返信後の記録

- 監査ログ（下記）に記録し、`docs/automation/logs/` に保存
- 月次で「口コミ数・平均・返信率・返信リードタイム」を BI AI へ連携（13_KPI C-1/C-2）
- 分類 C の要望内容は月次レポートの改善提案の入力とする

## 炎上・クレーム時の手動切替

- 以下のいずれかで**フロー全体を即時 L3 化**（自動ドラフト生成も停止）：
  - 同一投稿者・関連投稿者からの連続ネガティブ投稿
  - SNS 等での拡散の兆候
  - 法的措置・行政（保健所等）への言及
- L3 化後は 07_SNS_AI・10_Legal_AI のエスカレーション手順に従い、対応方針は CEO が決定する

## kill switch

- CEO・Executive AI の停止指示でフロー即時停止（検知・ドラフト・承認待ちをすべて凍結）
- 実装：n8n ワークフロー無効化＋返信実行前の停止フラグ確認ノード（#001 と同方式の二重化）
- 復旧は CEO の再開指示のみ

## 監査ログ（全件記録・削除改変禁止）

| 項目 | 内容 |
|---|---|
| 口コミ | 取得日時・星数・本文・投稿者表示名 |
| 分類 | A〜E と判定理由 |
| ドラフト | 使用テンプレート・生成文・生成日時 |
| 審査 | Legal/Medical QA の関与有無と結果 |
| 承認 | 承認者・日時・承認版 |

## n8n 実装時のノード構成案

```
1. Schedule Trigger（毎日 08:30）
2. HTTP Request：GBP API で新規レビュー取得（前回実行以降）
3. IF：新規なし → 終了／あり → 続行
4. AI ノード（分類）：A〜E を構造化出力で判定（星数＋本文）
5. Switch ノード：
   - D・E → GitHub Issue（緊急・CEO_INPUT_REQUIRED）＋即時通知 → 終了（以降は手動）
   - A〜C → 続行
6. AI ノード（ドラフト生成）：承認済みテンプレートから選択・整形（新規文章生成なし）
7. Function：個人情報・症状言及・禁止表現の機械チェック → NG は Marketing AI 差し戻し Issue
8. GitHub ノード：日次承認ファイル生成（CEO_INPUT_REQUIRED・口コミ原文＋ドラフト一覧）
9. Wait：承認 Webhook（期限48時間）→ 超過は翌日再提示
10. IF：停止フラグ確認（kill switch）
11. Function：承認版と送信版の一致検証
12. HTTP Request：GBP API で返信投稿（Retry 3回 → 失敗は手動切替通知）
13. GitHub ノード：監査ログ書き込み・KPI 集計ファイル更新
```

- 運用初期12週間は #001 と同様、AI 分類・ドラフトに対する人の確認を並行し、乖離を記録する

## CEO 判断事項

2. 分類 A〜C の承認方式：日次一括で問題ないか（1件ずつ個別承認も選択可）
3. ネガティブ口コミ（D）発生時の第一連絡先・連絡手段の指定
