# フロー設計書 #001：GBP 投稿（自動化レベル L2）

作成：Automation AI ／ 作成日：2026-07-12 ／ 状態：設計案・CEO 承認待ち
自動化レベル：**L2（AI 下書き → CEO 承認 → 実行）**。承認記録なしに公開できない構造とする。

## 目的

週1本の Google ビジネスプロフィール投稿を、品質ゲート（Medical QA → Legal → CEO）を通した上で、人手の作業を「承認判断のみ」に圧縮して安定運用する。

## 対象業務

- GBP「最新情報」投稿の下書き生成 → 検証 → 承認 → 公開 → 記録
- 対象外：口コミ返信（別フロー #002 として設計予定）、GBP 基本情報の変更（手動・CEO 承認制）

## トリガー

- 定時：毎週月曜 09:00 に下書き準備を開始（投稿予定日：木曜）
- 手動：Executive AI がテーマ展開時に臨時実行を指示

## 入力情報

1. 投稿カレンダー（当週のテーマ・使用素材ファイル）
2. 承認済みテーマの Marketing AI 成果物（例：`docs/research/2026-07-ankle-cai/05-gbp-posts.md`）
3. `docs/meo/nap-template.md`（表記の正）／`docs/meo/gbp-info-sheet.md`
4. 禁止表現マスターリスト（`Yuki_AI_OS/10_Legal_AI.md`）

## フロー全体

```
[定時トリガー]
→ ① AI 下書き生成（Marketing AI 成果物から当週分を整形）
→ ② Research AI 連携（数値・引用の照合）
→ ③ Medical QA チェック（医学的正確性・安全性）
→ ④ Legal AI チェック（禁止表現・保険誤認・機械チェック併用）
→ ⑤ CEO 承認（承認 / 修正依頼 / 却下）
→ ⑥ 公開（承認記録の存在を検証してから実行）
→ ⑦ 記録（監査ログ・カレンダー更新・KPI 連携）
```

### ① AI 下書き生成

- 承認済み成果物ファイルから当週分の投稿文を取得し、文字数（300〜450字）・冒頭要点・CTA ボタン設定・写真の指定を整形
- 成果物にない新規文章を生成しない（**整形のみ**。新規作成が必要な場合は Marketing AI へ差し戻し）

### ② Research AI 連携

- 投稿文中の数値・医学的主張を「引用可能数値リスト」と照合
- リスト外の数値が検出された場合はフローを停止し、Research AI へ確認依頼

### ③ Medical QA チェック

- 検証結果票を発行（判定：承認/条件付き/差し戻し）
- 差し戻し時は指摘＋修正案を添えて①へ戻す（自動修正はしない）

### ④ Legal AI チェック

- 前段で禁止表現の機械チェック（L1・grep）を実行し、通過後に Legal AI が審査結果票を発行
- **機械チェック通過＝Legal 承認ではない**（両方の記録を必須とする）

### ⑤ CEO 承認フロー

- CEO へ承認依頼を通知（投稿文全文・写真・投稿予定日時・Medical QA/Legal の結果票を添付）
- 応答選択肢:「承認」「修正依頼（コメント付き）」「却下」
- 承認期限：投稿予定時刻の3時間前。期限までに応答がない場合は**公開せず翌週に繰り越し**（無応答＝承認とみなす設計は禁止）

### ⑥ 公開手順

1. 承認記録（承認者・日時・対象版のハッシュ）の存在を検証
2. 承認された版と公開する版が同一であることを検証（承認後の無断改変防止）
3. GBP API または管理画面への手動投稿（API 不通時のフォールバック）で公開
4. 公開後、実際の表示を取得してスクリーンショット保存

### ⑦ 公開後の記録

- 投稿カレンダーの状態を「公開済み」に更新
- 監査ログに記録（下記）
- 30日後に当該投稿の閲覧数を取得し、BI AI の月次データへ連携

## 失敗時の対応

| 失敗点 | 挙動 |
|---|---|
| 各チェックで差し戻し | ①へ戻す。同一投稿が2回差し戻された場合は Executive AI へエスカレーション |
| API エラー | 3回まで指数バックオフでリトライ → 失敗時は人へ通知し手動投稿に切替 |
| 承認期限切れ | 公開せず翌週繰り越し。CEO へ「繰り越し」を通知 |
| 公開後に誤り発見 | 即時に投稿を非公開化 → インシデント報告書を発行 → 修正版を①から再実行 |

## kill switch

- CEO・Executive AI のいずれかが停止指示を出すと、フロー全体が即時停止（生成中・承認待ち・公開予約のすべてを凍結）
- 実装：n8n のワークフロー無効化＋公開ステップ前の「停止フラグ」確認ノード（二重化）
- 復旧は CEO の再開指示のみで行う

## 監査ログ（全実行で記録）

| 項目 | 内容 |
|---|---|
| 実行 ID・日時 | フロー実行ごとに一意 |
| 入力 | 使用した成果物ファイルとその版（コミットハッシュ） |
| 各ゲートの結果 | Research 照合／Medical QA 票／Legal 票／機械チェック結果 |
| 承認 | 承認者・承認日時・承認対象版 |
| 公開 | 公開日時・投稿 URL・スクリーンショット |
| 異常 | リトライ回数・エラー内容・エスカレーション有無 |

保存先：`docs/automation/logs/`（月次でアーカイブ）。削除・改変禁止。

## n8n 実装時のノード構成案

```
1. Schedule Trigger（毎週月曜 09:00）
2. GitHub ノード：投稿カレンダー・成果物 md を取得
3. Function ノード：当週分の抽出・文字数/形式の整形（新規生成なし）
4. Function ノード：禁止表現 grep チェック（L1）→ NG なら Stop & Error
5. AI ノード（Anthropic API）×3：Research 照合 / Medical QA 検証票 / Legal 審査票の生成
   ※各票は「承認/差し戻し」を構造化出力（JSON）で返す
6. IF ノード：3票すべて承認か？ → No：GitHub Issue 作成（差し戻し内容）+ 通知して終了
7. 承認依頼ノード：LINE Notify / メールで CEO へ全文＋結果票を送信（Wait: Webhook 承認応答）
8. Wait ノード（承認期限まで）→ タイムアウト時：繰り越し処理へ分岐
9. IF ノード：停止フラグ確認（kill switch）
10. Function ノード：承認版ハッシュと公開版の一致検証
11. HTTP Request ノード：GBP へ投稿（失敗時 Retry 3回 → 手動切替通知）
12. HTTP Request ノード：公開確認・スクリーンショット取得
13. GitHub ノード：カレンダー更新・監査ログ書き込み
14. 通知ノード：CEO へ「公開完了」報告
```

- 認証情報はすべて n8n credentials に格納（フロー定義への直書き禁止)
- ステップ5の AI 検証は一次スクリーニングであり、運用初期（最初の12週間）は Medical QA・Legal の**人による確認**を並行させ、乖離がないことを確認してから AI 票のみに移行するか CEO が判断する

## CEO 判断事項

1. 本フロー設計の承認（特に承認期限3時間前・無応答は繰り越しのルール）
2. 承認通知の受信手段（LINE / メール / その他）
3. 運用初期12週間の並行確認後、AI 検証票のみへ移行するか
